# Repo Confidence Methodology

## 1. 问题定义

AI 编程工具解决了大量执行问题，却放大了一个新的组织风险：代码变化速度超过了人的理解速度。

开发者可能可以快速提交修改，但无法稳定回答：

- 为什么系统采用当前设计？
- 正常链路和失败链路分别是什么？
- 这次修改改变了哪些状态和约束？
- AI 是否遗漏了缓存、并发、迁移、权限或计费？
- 如何证明改动安全？
- 出现问题后如何定位和回滚？

Repo Confidence 将这个问题称为 **Ownership Gap**：能够生成修改，与能够理解、验证并负责之间的差距。

## 2. 目标状态

我们不把“阅读完所有代码”作为项目接手完成标准。一个可信的维护者应当具备：

1. **导航能力**：能快速定位入口、状态和责任模块；
2. **因果理解**：能解释关键链路为何这样工作；
3. **边界意识**：知道哪些内容已确认、哪些仍未知；
4. **影响判断**：能预测一个需求可能波及的区域；
5. **验证能力**：能设计测试、观察点和回滚路径；
6. **持续更新**：能把真实需求中的新知识沉淀回项目地图。

## 3. 三层知识模型

### 3.1 人需要内化的稳定知识

- 产品目标与业务词汇；
- 模块边界与依赖方向；
- 最重要的业务链路；
- 数据事实源和状态生命周期；
- 安全、权限、计费、一致性等系统不变量；
- 高风险区域和验证方式。

### 3.2 人需要能够快速查找的参考知识

- API、表、缓存 Key 和配置索引；
- Provider、任务、事件和错误码矩阵；
- 测试与实现的对应关系；
- 迁移和兼容性历史；
- 常见变更 Playbook。

### 3.3 交给 AI 按需恢复的细节

- 具体函数和字段；
- 单个供应商的协议差异；
- 历史兼容分支；
- 低频边界条件；
- 当前 Git 提交下的精确调用路径。

这个分层避免让人背代码，也避免完全依赖即时问答。

## 4. 核心工件

### 4.1 Project Passport

仓库的一页身份证，包括技术栈、启动方式、主要入口、依赖、构建测试命令、部署形态和关键约束。

### 4.2 Evidence Manifest

只读、机器可读的证据索引，用于连接入口、路由、后台任务、持久化模型、测试和
迁移候选。它先回答“应该从哪里开始核实”，再由 CodeGraph、源码、测试和运行
行为回答“系统实际上怎样工作”。

Evidence Manifest 中的静态注册、启发式关联和发现分数都不是业务事实，必须
保留候选状态，不能直接升级为 `Verified`。

### 4.3 Project Atlas

项目知识导航，连接业务域、实体、链路、状态、测试、风险和维护 Playbook。

### 4.4 Flow Card

一条业务链路的可维护性描述，必须同时覆盖：

- 正常路径；
- 失败、重试和补偿；
- 数据与缓存变化；
- 权限、计费和审计；
- 不变量；
- 测试与观测；
- 常见修改点。

### 4.5 Change Brief

在修改代码前生成的影响说明。它不是冗长的设计文档，而是帮助人和 AI 对齐范围、风险和验证方法的最小控制面。

### 4.6 Known Unknowns

显式记录当前无法确认的问题。它将“我不知道自己遗漏了什么”的恐惧，转化为有限、可追踪的学习和验证任务。

## 5. Confidence Loop

### Discover

使用仓库说明、构建文件、CodeGraph、搜索、测试和 Git 历史发现系统结构。

### Externalize

将发现压缩为地图、链路、不变量和未知清单。不要堆积源码摘要。

### Teach Back

让开发者脱离文档，用自己的语言解释系统。无法解释的区域重新进入发现阶段。

### Simulate

给出一个需求或故障，让开发者预测入口、影响、失败模式和验证方案。

### Change

由开发者与 AI 共同完成真实、范围明确的修改。

### Verify

运行测试、观察日志和状态，检查实际行为是否符合预测。

### Refresh

将新发现的不变量、风险和链路补回知识库，并记录验证提交。

## 6. 证据模型

每条重要知识应包含：

```text
Claim       结论
Status      Verified / Inferred / Unknown
Evidence    源码符号、测试、迁移、配置或运行输出
Commit      最近验证提交
Risk        结论错误或过期的影响
```

### Verified

当前代码、测试或运行行为直接支持该结论。

### Inferred

结论合理但缺少完整证明。例如只看到写入逻辑，没有确认所有读取方。

### Unknown

证据不足或存在冲突。不要用流畅语言掩盖未知。

## 7. 新人训练协议

### 阶段一：导航

- 运行项目和测试；
- 复述项目目标；
- 定位主要入口；
- 解释模块职责；
- 使用工具追踪一条调用链。

### 阶段二：理解

- 学习核心实体；
- 学习五到八条关键链路；
- 描述数据与缓存状态变化；
- 识别系统不变量和高风险区。

### 阶段三：推演

- 对真实需求生成影响预测；
- 对常见故障推演失败和恢复路径；
- 比较自己的预测与 AI/代码证据。

### 阶段四：贡献

- 完成低风险修改；
- 解释 Diff；
- 编写或更新测试；
- 将新知识沉淀回 Atlas。

### 阶段五：维护

- 修改核心链路；
- 处理线上问题；
- 设计灰度和回滚；
- 审查 AI 或他人的高风险变更。

## 8. 需求处理协议

默认交付模式：

```text
需求
→ AI 生成 Change Brief
→ 在 commentary 中说明关键影响
→ AI 继续实施
→ 测试和验证
→ 总结 Diff 与残余风险
```

显式学习模式：

```text
需求
→ AI 生成 Change Brief 但暂停修改
→ 开发者预测修改点
→ AI 给出证据和反馈
→ 实施修改
→ 开发者复述影响和验证方法
```

学习模式必须由用户明确开启，避免不必要地阻断正常交付。

## 9. 知识更新策略

- 每份核心文档记录最近验证提交；
- 优先引用稳定符号名，而不是只引用行号；
- 使用 Git Diff 判断哪些链路可能过期；
- 生成内容和人工确认内容明确分区；
- PR 修改关键模块时检查相关 Flow Card；
- 不要求每个代码变化都改文档，只维护稳定知识和关键约束。

## 10. 反模式

### 自动生成几百页代码摘要

信息量很大，但没有形成因果模型，且很快过期。

### AI 修改后才解释

开发者失去预测和纠错机会，长期能力不会增长。

### 把 CodeGraph 当成唯一事实源

调用关系无法自动表达业务原因、历史约束、运行状态和团队决策。

### 以函数名记忆作为考核

真正需要考核的是状态变化、失败语义、风险判断和验证方法。

### 追求所有领域全部变绿

维护者应当能界定风险并按需求深入，不需要在接第一个需求前理解全部代码。

## 11. 成功指标

Repo Confidence 不以生成文档数量为成功标准。更有意义的指标是：

- 新人定位需求入口所需时间；
- Change Brief 对真实修改范围的覆盖率；
- 新人对关键链路的复述准确度；
- 低风险需求独立完成率；
- AI 改动中缓存、迁移、权限、计费遗漏数量；
- 线上故障定位和恢复时间；
- Project Atlas 的验证新鲜度；
- 团队已知未知的收敛速度。
