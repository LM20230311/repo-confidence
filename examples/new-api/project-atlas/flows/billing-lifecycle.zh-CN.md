## 核心心智模型

计费是一条请求生命周期，不是请求结束后顺便写一条日志。系统先确定定价模式
和资金来源，创建一个 `BillingSession`，同时预扣 Token 额度与钱包或订阅
额度，然后根据实际 Usage 结算差额；请求失败则退款。

## 三种定价模式

| 模式 | 输入 | 实际结算依据 |
|---|---|---|
| 传统倍率 | 模型、输出、缓存、图片、音频和分组倍率 | 归一化后的实际 Usage × 各类倍率 |
| 固定价格/按次 | 模型价格与可选任务倍率 | 请求或任务数量 |
| 阶梯表达式 | 带版本的 Token/请求变量表达式 | 用实际 Usage 重新执行冻结的表达式 |

阶梯表达式快照会保存表达式文本、哈希、版本、分组倍率、预估 Token 和额度
换算参数。结算使用请求开始时冻结的输入，而不是管理员后来修改的新配置。

## 正常路径

1. 预估输入 Token 和最大输出范围。
2. 确定模型价格或倍率、分组倍率和各专项倍率。
3. 根据 `subscription_only`、`wallet_only`、`wallet_first` 或 `subscription_first` 创建 `BillingSession`。
4. 预扣 Token 额度。
5. 预扣选中的钱包或订阅额度；失败时尝试回滚 Token 预扣。
6. 调用上游供应商。
7. 归一化实际 Usage，包括缓存、图片和音频类别。
8. 使用冻结的价格上下文计算实际额度。
9. 对资金来源和 Token 额度结算 `actual - preConsumed`。
10. 写入消费和审计明细，包括额度饱和标记。

## 失败与补偿

- 资金预扣成功前失败：回滚已经完成的 Token 预扣。
- 已预扣但尚未结算时失败：延迟错误处理调用 `Refund`。
- `BillingSession` 使用互斥锁和状态字段防止重复结算或退款。
- 订阅退款使用 Request ID 标识的持久化预扣记录，因此可以重试。
- 钱包退款是 `quota += N` 的非幂等操作，为避免多退而不自动重试。
- 如果资金来源已经结算成功，但 Token 差额调整失败，会记录错误并把会话标记为已结算，不会错误退还已经提交的资金。

## 安全控制

- 拒绝负数额度输入。
- 用户可控倍率在计算前限制范围。
- 额度换算使用饱和辅助函数，避免未检查的浮点数直接转换为整数。
- 过滤 NaN、无穷大和非正数任务倍率。
- 额度饱和信息写入管理员可见的日志元数据。
- 异步任务强制全额预扣，不能使用信任额度旁路。

## 不变量

- 预扣和结算必须使用同一份冻结价格契约。
- 溢出或非法倍率不能产生负数扣费。
- 成功结算时，钱包/订阅和 Token 额度必须按相同方向变化。
- 一个失败请求在当前活动会话中最多退款一次。
- 订阅预扣必须能通过 Request ID 幂等识别。
