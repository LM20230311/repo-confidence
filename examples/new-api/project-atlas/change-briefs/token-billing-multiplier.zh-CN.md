# 改动影响单：Token 专属计费倍率

> 分析提交：[`4e570389`](https://github.com/QuantumNous/new-api/tree/4e570389dd433a717373ce9c9b822b59f5ed3d5d)
> 风险：高
> 状态：仅完成影响分析，没有修改 New API 上游代码
>
> [English version](token-billing-multiplier.md)

## 目标和验收标准

为 API Token 增加可选倍率，使同一个用户的两枚 Token 可以采用不同收费标准，
同时不修改模型或分组价格配置。

验收标准：

- 历史 Token 等价于倍率 `1`，行为保持不变；
- 倍率必须校验范围，不能是 NaN、无穷大或非法值；
- 完整应收费金额只能应用一次倍率；
- 传统倍率、固定价格、阶梯表达式、音频和异步任务结果一致；
- 预扣和实际结算使用同一个请求开始时的倍率；
- 后续修改 Token 不会改变进行中的请求或任务价格；
- 钱包或订阅额度与 Token 额度按相同的最终额度结算；
- 用量日志保存倍率和最终有效额度；
- 溢出继续使用已有的饱和换算和审计路径。

不在本次范围：

- 替换模型、输出、缓存或分组倍率；
- 修改渠道选择；
- 新增货币或支付供应商；
- 直接在 New API 上游仓库实现该功能。

## 当前行为和证据

### Token 持久化与缓存——已验证

`model.Token` 保存额度、模型限制、IP 限制、分组和跨组重试，没有价格调整
字段。`AddToken`、`UpdateToken` 和 `Token.Update` 会显式复制或选择允许修改
的字段。Redis 使用 Key 的 HMAC 保存 Token 对象。鉴权把选中的字段写入 Gin
Context，再由 `RelayInfo` 带入后续链路。见 [E18](../evidence.md)。

### 计价——已验证

`PriceData` 包含模型、输出、缓存、图片、音频和分组倍率，以及供应商或请求
专用的 `otherRatios`。传统倍率、固定价格和阶梯表达式拥有不同的预扣与实际
结算路径；音频和异步任务还有专项路径。见
[E10、E11、E19](../evidence.md)。

### 异步任务——已验证

`TaskBillingContext` 保存模型价格、模型倍率、分组倍率和 `otherRatios`，让任务
离开原始请求后仍能在轮询阶段结算。它目前没有通用商业倍率快照。见
[E20](../evidence.md)。

## 建议的业务语义

使用一个明确的最终倍率：

```text
基础额度 = 模型和组件计价 + 应参与计费的附加费
商业额度 = 基础额度 × Token 计费倍率
```

规则：

- 默认值必须精确等于 `1`；
- `0` 不代表免费，免费仍然是明确的价格策略；
- 产品需要规定上限，例如 `(0, 1000]`；
- 在组件价格和供应商/请求专项倍率之后应用；
- 在最终边界只执行一次受检查的 Decimal 到 Quota 换算；
- 请求开始时的倍率用 Decimal 字符串或等价的精确格式保存到快照。

倍率上限以及普通用户能否修改自己的倍率属于产品决策。它们会改变商业和滥用
风险，因此在实现前必须明确。

## 为什么不能直接使用 `otherRatios["token"]`

这个捷径看起来可行，因为普通文本结算会应用 `PriceData.otherRatios`，但它并不
完整：

1. 阶梯表达式结算不会调用 `ApplyOtherRatiosToDecimal`；
2. 音频结算使用独立公式；
3. 异步 Adaptor 可能调用 `ReplaceOtherRatios`，导致 Token 倍率丢失；
4. 这张 Map 表达的是时长、尺寸、生成数量等供应商或请求维度；
5. 日志和历史审计没有稳定的商业倍率字段。

这说明数据结构看起来相似，不代表业务语义相同。

## 实施方案

### 1. 定义并校验持久化字段

- 为 Token 增加倍率字段，数据库默认值等价于 `1`。
- 显式更新创建逻辑、更新字段列表和返回结构。
- 拒绝无权限修改、非正数、NaN、无穷大和超过产品上限的值。
- 明确是否只有管理员可以设置，不能根据“用户拥有 Token”推断权限。
- 验证 SQLite、MySQL 和 PostgreSQL 的迁移行为。

### 2. 创建请求级不可变状态

- 把倍率写入 Token 鉴权上下文。
- 在生成 `RelayInfo` 时只复制一次。
- 在这个边界把历史空值归一化为 `1`。
- 结算时不能重新读取 Token 表获取可能已经变化的新倍率。

### 3. 引入一个明确的最终倍率函数

语义类似：

```text
ApplyCommercialMultiplier(基础 Decimal 金额, 快照倍率)
→ 受检查、可饱和的整数 Quota + 审计标记
```

不要在 `BillingSession` 中应用倍率；会话应继续与价格构成解耦。

### 4. 接入所有定价模式

- 传统倍率和固定价格请求的预扣预估要乘倍率。
- 阶梯计费快照新增倍率，并在预估和实际结算时使用。
- 按业务契约，在工具、搜索、图片和音频附加费之后应用倍率。
- 音频、Realtime 和其他专项额度计算统一走最终倍率函数。
- 明确测试免费模型和 Usage 为零时的行为。

### 5. 持久化异步快照

- 在 `TaskBillingContext` 中保存倍率。
- 读取没有该字段的旧快照时默认使用 `1`。
- 提交阶段调整和任务完成重算都只能应用一次。
- Adaptor 替换 `otherRatios` 时不能改变商业倍率。

### 6. 让每一笔费用可以解释

日志至少保存：

- 应用 Token 倍率前的基础额度；
- Token 倍率快照；
- 最终额度；
- 定价模式和已有模型/分组维度；
- 发生额度饱和时的审计标记。

历史账单不能依赖 Token 当前的倍率值。

## 失败和并发分析

- Token 修改与请求并发时，请求使用鉴权阶段读取到的值，并在后续保持该快照。
- Token 更新后 Redis 可能短暂返回旧值；这属于现有缓存一致性边界，需要测试和说明。
- 倍率增加可能使预扣超过钱包、订阅或 Token 额度；能预估时必须在调用上游前拒绝。
- 倍率溢出必须产生可见的饱和标记，不能绕回负数扣费。
- 退款期间进程崩溃仍是已有的持久化缺口，新增倍率不会自动解决它。
- 部署前创建的异步任务没有倍率字段，必须按 `1` 读取。

## 必须保持的不变量

- 倍率只能应用一次。
- 预扣和结算使用同一份不可变倍率快照。
- Token 额度和选中的钱包或订阅按相同最终额度变化。
- 修改倍率不能重写历史账单语义。
- 供应商专项倍率不能删除或替换 Token 倍率。
- 倍率为 `1` 时，在舍入允许范围内必须保持历史行为。

## 验证计划

### 计价矩阵

使用 `0.5`、`1`、`2` 和产品最大倍率测试：

- 包含输入、输出和缓存的传统文本计费；
- 固定价格请求；
- 发生档位切换的阶梯表达式；
- 工具或搜索附加费；
- 音频输入和输出；
- 异步按次任务；
- 根据真实时长或 Token 重算的异步任务。

### 生命周期矩阵

- 钱包和订阅资金来源；
- 预扣额度不足；
- 实际额度高于或低于预估；
- 上游失败和退款；
- 重复结算和重复退款；
- 预扣后、结算前修改 Token；
- 异步任务完成前进程重启；
- Redis 命中、未命中和旧值窗口。

### 兼容和审计

- 在三种数据库上迁移历史 Token；
- 读取没有倍率字段的旧任务快照；
- 断言日志包含基础额度、倍率和最终额度；
- 断言额度饱和出现在管理员元数据；
- 断言倍率为 `1` 时普通模型和分组计价不变。

## 灰度和回滚

1. 先部署默认倍率为 `1` 的字段和兼容旧快照的读取逻辑。
2. 增加影子计算，只比较新旧额度，不按新结果扣费。
3. 使用功能开关只允许管理员配置。
4. 按定价模式对比预扣、结算和日志对账。
5. 先开放少量 Token，再逐步扩大。

回滚时停止对新请求应用倍率，但必须继续保留字段和快照读取能力，直到新契约下
创建的所有异步任务都进入终态。

## 实现前必须确认的问题

| 问题 | 影响 | 是否阻塞 | 决策来源 |
|---|---|---:|---|
| 谁可以设置倍率？ | 防止用户给自己打折 | 是 | 产品与安全决策 |
| 是否允许小于 `1`？ | 定义折扣语义 | 是 | 商业决策 |
| 哪些附加费参与倍率？ | 直接影响收入 | 是 | 明确计费契约 |
| 数值范围和精度是多少？ | 影响存储和溢出 | 是 | 财务与工程决策 |
| 订阅套餐消耗倍率前还是倍率后额度？ | 改变权益消耗 | 是 | 产品决策 |
| 违规费用是否应用倍率？ | 影响滥用成本 | 是 | 安全与产品决策 |

## 风险结论

高风险。它同时影响鉴权产生的请求状态、数据库迁移、Redis 缓存、所有定价
模式、钱包和订阅余额、Token 额度、异步快照以及财务审计。实现速度快并不会
降低这些风险。
