# Contributing to Repo Confidence

感谢你愿意参与 Repo Confidence。

这个项目关注的不是单纯提升 AI 写代码的速度，而是帮助开发者和团队建立对陌生仓库、代码修改与长期维护责任的真实信心。

## 可以贡献什么

- 改进项目接手和维护方法论；
- 提交真实但已脱敏的 onboarding 案例；
- 为 Go、Java、Python、JavaScript、Rust 等生态补充专项分析策略；
- 改进 Project Atlas、Flow Card、Change Brief 模板；
- 增加 Codex Skill、其他 Agent 或 IDE 集成；
- 提供训练题、故障推演和能力评估方法；
- 修复脚本、文档和可用性问题；
- 提交失败案例，指出方法在什么场景下不起作用。

## 贡献原则

### 证据优先

涉及仓库行为的结论应尽量关联源码、测试、迁移、配置或可复现输出。不要把推断写成确定事实。

### 保护隐私

不要提交公司私有源码、密钥、客户信息、生产地址或未授权文档。示例必须使用公开仓库、合成数据或充分脱敏的材料。

### 保持通用

核心方法应能跨语言和框架复用。框架专项内容放在独立参考资料中，不要让主流程依赖某一种技术栈。

### 让人参与理解

新增流程不应只是让 AI 自动生成更多文档。请说明它如何帮助人形成心智模型、校准未知、推演风险或验证改动。

## 提交流程

1. 创建 Issue 描述问题、使用场景或提案；
2. 将变更保持在清晰、可审查的范围内；
3. 为脚本增加测试或提供可复现验证步骤；
4. 更新受影响的文档、模板和 Skill 指令；
5. 在 Pull Request 中说明：
   - 解决了什么问题；
   - 对新人或维护者有什么帮助；
   - 如何验证；
   - 有哪些限制和未解决问题。

## 本地验证

验证仓库盘点脚本：

```bash
python3 skills/repository-onboarding-coach/scripts/repo_inventory.py .
```

验证 Skill 结构：

```bash
python3 /path/to/skill-creator/scripts/quick_validate.py \
  skills/repository-onboarding-coach
```

## 提案建议

较大的方法论改动建议先回答：

- 它解决的是信息获取、长期理解、改动判断还是维护训练？
- 开发者使用后会获得什么可观察的能力提升？
- 如何避免生成过时或无法验证的知识？
- 是否增加了不必要的流程负担？
- 能否在真实项目和真实需求中验证？

我们欢迎不同观点。Repo Confidence 本身也应该通过真实使用不断被验证和修正。
